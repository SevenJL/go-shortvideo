## 2. 被测接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/healthz` | GET | 健康检查（无业务逻辑，作性能基线） |
| `/api/videos/{id}/like` | POST | 点赞（幂等），需 `X-User-Id` 头 |

点赞接口有两种路径：
- **幂等路径**：用户已点过赞，请求直接返回（读检查 + 提前返回）
- **写路径**：用户首次点赞，需更新 `userLikes` map 和视频 `LikeCount`

**Apache Bench 交叉验证 (幂等路径)**:

| 并发 | 请求数 | QPS (ab) | 平均延迟 | 错误数 |

## 5. 性能分析

### 5.1 QPS-并发曲线

```
QPS
60k │         ●────●────●────●
50k │    ●
40k │
30k │
20k │  ●
10k │
  0  ├─────┬─────┬─────┬─────┬─────► 并发
      1    10    50   100   200
```

**关键观察**:
- 并发 1→10：QPS 从 16k 跃升至 56k（**3.5× 提升**），Go HTTP server 的多 goroutine 调度充分展开
- 并发 10→200：QPS 稳定在 56k–60k，**已触及天花板**
- 天花板原因：`sync.RWMutex` 的 `Lock()` 将所有写操作串行化

### 5.2 延迟分布特征

```
幂等路径 (c=50):
  P50:   112µs  ┤
  P75:   119µs  ┤
  P99:   124µs  ┤
  Avg:   851µs  ┤████████░░░░░░░░  (被少量等待拖高)
  Max:  12.3ms  ┤██
```

**延迟呈双峰分布**：
- **快路径 (>99%)**: ~100µs 完成——纯内存 map 查找，无锁竞争
- **慢路径 (<1%)**: 毫秒级——等待 `RWMutex.Lock()` 释放

### 5.3 瓶颈诊断

| 瓶颈层 | 组件 | 影响 |
|--------|------|------|
| **🔴 主要** | `store.Store.mu` (sync.RWMutex) | 所有 Like 操作竞争同一把写锁 |
| 🟡 次要 | Go HTTP server goroutine 调度 | c=200 时出现 80ms 尾部延迟 |
| 🟢 无影响 | JSON 序列化、GC | 均在微秒级，不构成瓶颈 |

**当前架构简图**：
```
HTTP goroutines (N个)
    ↓ 全部排队
sync.RWMutex.Lock()
    ↓ 串行
map 查找 + map 写入
    ↓
返回 JSON
```

## 6. 结论与建议

### 6.1 v0.1.0 性能总结

| 指标 | 数值 |
|------|------|
| **点赞 QPS (幂等)** | **~58,000/s** |
| **点赞 QPS (写路径)** | **~50,000/s (5 实例聚合)** |
| **P99 延迟 (c≤200)** | < 250µs |
| **错误率** | **0%** |
| **内存占用** | ~19 MB |
| **稳定性** | ✅ 全部测试零错误 |

### 6.2 适用场景

- ✅ 中小规模短视频应用（DAU < 50 万）
- ✅ 开发 / 测试环境
- ✅ 演示和原型验证
- ⚠️ 生产环境需替换为 MySQL + Redis（store 层接口已预留）

### 6.3 后续优化方向

1. **分片锁**: 按 `videoID % N` 将锁拆分为 N 个分片，降低竞争
2. **异步写**: 点赞事件先入 MQ，Worker 异步批量写库（`cmd/likeworker` 已预留架构）
3. **Redis 替换**: 启用 `pkg/redisx`，将热点视频计数放在 Redis Hash 中
4. **连接池**: 使用 `fasthttp` 替代 `net/http` 可提升 ~20% QPS

### 6.4 与设计文档对照

| 设计目标 | 当前实现 | 状态 |
|----------|----------|------|
| 点赞去重按用户维度 | `userLikes[userID][videoID]` map | ✅ |
| 推拉结合关注流 | `feed` 包 + `fanout` worker | ✅ 已实现，待压测 |
| 高并发点赞 | sync.RWMutex（单锁） | ⚠️ 需分片/异步化 |
| 关注流写扩散 | `ListFollowers` + fanout worker | ✅ 架构就绪 |
