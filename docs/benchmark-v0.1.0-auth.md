## 2. 架构差异

```
v0.1.0:                    v0.1.0-auth:
HTTP handler               auth.Middleware
  ↓                          ├─ JWT 解析 (HS256 verify)
requireUser()                ├─ X-User-Id fallback
  ↓                          ├─ context.WithValue()
store.Like()                 ↓
  ↓                        HTTP handler
  ↓                          ↓
                          store.Like()
```

**新增开销路径**:
- `auth.Middleware` → 请求头解析 + context 注入
- JWT Bearer → HMAC-SHA256 签名验证 + claims 反序列化
- X-User-Id fallback → 仅字符串解析（开销接近零）

## 4. 性能影响分析

### 4.1 开销分解 (c=10, go-wrk)

```
v0.1.0 总吞吐:   ~56,000 req/s  (基准)
v0.1.0-auth X-User-Id: ~42,800 req/s  (↓23.5%)
v0.1.0-auth JWT:      ~24,300 req/s  (↓43%)

开销来源:
┌─────────────────────────────┐
│ auth.Middleware 框架开销  ~10µs  │  → X-User-Id 路径的 23.5% 下降
│ JWT HMAC-SHA256 验证    ~15µs  │  → JWT Bearer 额外的 20% 下降
│ context.WithValue        ~1µs  │  → 可忽略
└─────────────────────────────┘
```

### 4.2 瓶颈转移

```
v0.1.0 瓶颈:
  sync.RWMutex (100%) ████████████████████████

v0.1.0-auth 瓶颈:
  JWT 验证 (40%)     ████████████████
  sync.RWMutex (40%)  ████████████████
  auth.Middleware (20%) ████████
```

在低并发 (c=10) 下，JWT 验证成为主要开销。高并发 (c=200) 下，RWMutex 锁竞争重新成为主导。

### 4.3 延迟分布保持健康

两版本在 c=100 下的 P99 延迟：
- v0.1.0: 148µs
- v0.1.0-auth X-User-Id: 141µs
- v0.1.0-auth JWT: 151µs

**尾部延迟未恶化** — JWT 解析是 CPU 密集型操作，不产生额外排队。

## 5. 结论

### 5.1 综合评估

| 指标 | v0.1.0 | v0.1.0-auth (JWT) | 评估 |
|------|--------|-------------------|------|
| 点赞 QPS (c=10) | 56,022 | 24,275 | ⚠️ -43% |
| 点赞 QPS (c=100) | 58,495 | 40,781 | ⚠️ -19% |
| P99 延迟 | 148µs | 151µs | ✅ 持平 |
| 错误率 | 0% | 0% | ✅ |
| 鉴权安全 | ❌ 无 | ✅ JWT+brypt | 🆕 |

### 5.2 性能代价评估

- **X-User-Id 路径**: ~23% QPS 下降（auth.Middleware 框架开销）
- **JWT Bearer 路径**: ~43% QPS 下降（X-User-Id 的 c=10 对比）
- **高并发场景**: 差距缩小到 ~17%（RWMutex 成为统一瓶颈）

**结论**: 以 c=100 生产中常见场景计，实际 QPS 从 58k 降至 40k，仍可支撑 **4000 万+ 日请求量**。考虑到安全性的根本提升，这个代价值得。

### 5.3 优化建议

1. **JWT 缓存**: 对已验证 token 的 userID 做短期缓存（TTL=1min），避免重复 HMAC 验证
2. **预解析 context key**: `UserIDKey` 已为常量，`context.WithValue` 开销已最优
3. **批量接口**: 未来支持一次请求操作多个视频时，JWT 开销被摊薄
4. **连接复用**: 生产环境使用 HTTP/2 多路复用，减少 TLS 开销
