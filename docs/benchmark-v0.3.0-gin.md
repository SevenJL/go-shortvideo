## 5. 性能提升分析

### 5.1 为什么 Gin 比 net/http 快 40%？

| 优化点 | net/http | Gin | 收益 |
|--------|----------|-----|------|
| JSON 序列化 | `encoding/json` | sonic (SIMD加速) | ~2x JSON 吞吐 |
| 路由匹配 | O(n) 遍历 | Radix Tree O(log n) | 大量路由时显著 |
| 内存分配 | 每次请求多次 alloc | 对象池复用 | GC 压力降低 |
| HTTP 解析 | 标准 net/http | 复用 net/http (无变化) | — |
| Context 传递 | `context.WithValue` alloc | `c.Set/c.Get` map | 减少 alloc |

### 5.2 版本演进总览

```
                      Healthz QPS    Like JWT c=100    Write 5x5
v0.1.0 (net/http)      54,818          —               50,066
v0.1.0-auth (JWT)      42,734         40,781            —
v0.2.0-prod (全栈)      —             33,048           39,746
v0.3.0-gin (JWT)       76,625         77,719           69,552
                       ↑ +40%         ↑ +91%            ↑ +75% (vs v0.2.0)
```

**v0.3.0-gin 的 JWT 路径 QPS (77,719) 甚至超过了 v0.1.0 的 X-User-Id 无鉴权路径 (58,495)** — Gin + sonic 的性能提升完全覆盖了 JWT 验证 + metrics + ratelimit 的全部开销。
